Système de régie live - CEC

Reverse engineering d'un protocole propriétaire, bridge réseau temps réel et pilotage caméra pour une équipe de captation réduite à une seule personne.

Rôle · Conception et développement, Individuel
Stack · Python, TCP/IP, ONVIF/HTTP, OBS Studio
Statut · En production depuis 2025

Contexte

Le CEC diffuse ses cultes en direct chaque semaine. La régie reposait sur du matériel et des logiciels disparates, opérés manuellement par plusieurs bénévoles en même temps - un fonctionnement coûteux en monde et fragile le jour J.

Problème & solution

Problème

Diffuser un culte demandait jusqu'à trois opérateurs distincts : un sur les caméras, un sur le logiciel de présentation des chants et versets, un sur la diffusion. Ce logiciel n'expose aucune API publique, ce qui empêchait toute synchronisation automatique avec l'overlay du flux vidéo.

Solution construite

Un système intégré et open-source : un script Python pilote trois caméras PTZ depuis une manette de jeu, et un bridge réseau lit en temps réel les données du logiciel de présentation pour les transmettre à la régie sous forme d'overlay synchronisé. Un seul opérateur suffit désormais.

Architecture du système

Trois caméras PTZ remontent leur flux vidéo en RTSP vers un switch PoE+, cœur du réseau local. Depuis ce même switch, un script Python pilote les caméras en ONVIF/TCP à partir d'une manette de jeu, tandis qu'un bridge maison expose en TCP/JSON sur le port 50505 les données lues dans le logiciel de présentation pour les réinjecter en overlay dans OBS. La diffusion sort en RTMP vers YouTube, l'audio passe par une console numérique puis une interface USB-C, et l'image est répartie sur plusieurs écrans via un splitter HDMI.

📋 Version simplifiée - lecture facile

Ce schéma décrit comment les caméras, les ordinateurs et les logiciels sont connectés ensemble pendant une diffusion live.

  1. 3 caméras filment la scène et envoient leur image en direct via le réseau (protocole RTSP).
  2. Un switch réseau (boîte avec des câbles) reçoit toutes les images et les distribue aux bons appareils.
  3. Un laptop reçoit les images, fait tourner EasyWorship 7 (logiciel de présentation) et OBS Studio (logiciel de diffusion).
  4. Un script Python lit en temps réel ce qu'affiche EasyWorship et envoie l'info à OBS pour créer un sous-titre automatique à l'écran.
  5. Une manette de jeu permet de piloter les caméras à distance via un autre script Python (protocole ONVIF).
  6. La console audio Behringer X32 gère le son, relié à une interface Focusrite par câble XLR.
  7. La vidéo finale part en direct sur YouTube via le protocole RTMP. Un splitter HDMI envoie l'image sur 3 écrans dans la salle en même temps.

Résultat : 1 seule personne suffit pour gérer toute la régie, contre 3 avant ce système.

Démarche technique

01
Reverse engineering
Capture réseau du trafic du logiciel de présentation pour identifier la structure des messages échangés à chaque changement de diapositive.
02
Développement du bridge
Écriture d'un serveur Python qui décode les trames binaires reçues et les expose en JSON sur le réseau local.
03
Intégration & contrôle
Overlay temps réel dans l'outil de diffusion, et pilotage des caméras par script Python relié à une manette.

Ce que ce projet démontre

Travailler avec un protocole propriétaire sans documentation officielle, livrer un système fiable dans un contexte où l'erreur n'est pas une option - un culte en direct ne s'interrompt pas - et concevoir une solution centrée sur l'opérateur qui l'utilise au quotidien plutôt que sur la seule prouesse technique.
